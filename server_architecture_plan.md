# サーバー構築・運用計画書 (Ubuntu + RTX 3070 Ti + Docker 統合管理)

---

## 0. ハードウェアリソース割当方針

> **方針:** 具体的なスレッド番号（`cpuset-cpus` の値）は、各サービスの `docker-compose.yml` を作成する段階で負荷特性を見ながら決定する。ここでは割当の基本方針のみを定める。追加サービスが増えるたびに柔軟に変更できる運用とする。

| コアの種類 | 担当する処理の方針 |
| :--- | :--- |
| **Pコア** (Thread 0-11) | 高負荷・リアルタイム性が命の処理（ゲームサーバー、映像配信・エンコード等） |
| **Eコア** (Thread 12-15) | バックグラウンド・軽負荷処理（Bot、同期処理、補助サービス等） |
| **未割当（デフォルト）** | 新規追加サービスはまず割当なしでデプロイ → 負荷状況を見て随時 `cpuset` を追記 |
| **GPU (RTX 3070 Ti)** | 映像エンコード（NVENC）・AI処理が必要なコンテナのみ `--gpus all` を付与 |
| **メモリ** | 各コンテナに `mem_limit` で上限を設定（Composeファイルで個別定義） |

## 1. システム要件・大前提

本システムの最大目的（大前提）は、**「スマートフォンに接続したARグラス（RayNeo Air 4 Pro等）で、Netflix等を1080p・無遅延で快適に視聴できること」** です。

加えて、12600KF（内蔵GPUなし）＋ RTX 3070 Ti（1枚）というハードウェア制約の下、マイクラサーバー・AIエンジン・Discord Bot等の複数サービスを安全に分離稼働させます。

---

## 2. アーキテクチャ方針

Proxmox等のハイパーバイザ・LXCを**採用しない**理由：
- 内蔵GPUが存在しないため、VMへのGPUパススルー時に物理コンソール（画面）が失われるリスクがある。
- LXC + Docker のネスト構成は cgroup・AppArmor の権限設定が複雑でメンテコストが高い。

**採用方針：Ubuntu 24.04 LTS（ベアメタル）+ Docker Compose による完全コンテナ分離**

---

## 3. ホストOS層（ベアメタルインフラ）

*   **OS:** Ubuntu 24.04 LTS（物理マシンへ直接インストール）
*   **役割:** コンテナを稼働させるための土台のみ。アプリは原則インストールしない。
*   **必須導入パッケージ（ホストに直接導入するもの）:**
    *   `NVIDIA Driver 550+`（RTX 3070 Ti 駆動用）
    *   `Docker Engine` + `Docker Compose Plugin`
    *   `NVIDIA Container Toolkit`（コンテナへのGPUパスに必要）
    *   `Tailscale`（ホスト側でVPNメッシュを張り、ストリーミングコンテナに `host` ネットワークで共有）

---

## 4. コントロール・統合管理システム層

### 管理ツール：`Dockge`（採用確定）
- `docker-compose.yml` スタック単位での管理に特化。軽量かつ直感的なUI。
- Portainerは高機能だが今回の用途（複数Composeスタックの管理）にはDockgeが最適。
- ブラウザから全コンテナの作成・起動・停止・ログ閲覧・リソース変更をGUI操作で完結。

### セキュアアクセス：`cloudflared`（Cloudflare Tunnel）
- **役割:** Dockge・各種WebUI等への外部からのアクセス口となるリバースプロキシ（受付窓口）。
- 動的IPでも固定FQDN（カスタムドメイン）でアクセス可能。ポート開放一切不要。
- Cloudflare Accessと組み合わせてGoogle認証（ゼロトラスト）を適用。自分以外はUIの画面にすら辿り着けない構成とする。

## 5. アプリケーション層（各機能コンテナ）

### 5.1. 映像配信コンテナ【大前提の実現】

*   **ベースイメージ:** `lizardbyte/sunshine` の公式Dockerイメージをベースに、Xvfb・Chrome・拡張機能を追加した **カスタムDockerfile** で構築（Ubuntu 24.04ベース）。
*   **内部構成:**
    *   `Xvfb`（仮想ディスプレイ、1920×1080）+ `PulseAudio`（仮想音声）
    *   Google Chrome：ハードウェアアクセラレーションOFF設定 ＋ 「Netflix 1080p」系拡張機能を組み込み（Linux特有のDRM画質制限 = 720p制限に対する回避策を適用）
    *   Sunshine：Xvfbの仮想画面をキャプチャし、RTX 3070 Ti の **NVENC** でH.265エンコードして配信
*   **リソース割当:**
    *   CPU: `cpuset-cpus: "8-11"`（Pコア4スレッド確保）
    *   GPU: `--gpus all`（NVENCエンコード用）
*   **ネットワーク:** `network_mode: "host"` を使用。ホストが持つTailscaleのP2P経路を直接利用し、スマホ・ARグラスとの間で極限低遅延のUDP通信（Moonlight）を確立する。
    *   ※ `host` ネットワークは分離性を下げるトレードオフがあるため、低遅延要件を優先して意図的に採用する（セキュリティ境界が弱まることを受容）。

#### 5.1.1. Netflix 1080p 回避策の運用前提（2026年3月時点）

*   **拡張機能の更新状況（Cadmium対応）**
    *   Firefox用 `Netflix 1080p UA`：`v1.0`（2026-02-25 更新）
    *   Chrome用 `Netflix - higher quality [QVI]`：`v2.0.95`（2026-02-24 更新）
*   **想定する動作原理:**
    *   Linux上でNetflixが720p上限になりやすい制約に対し、`Opera` へのUA偽装または再生プロファイル配列の書き換えで1080pストリームを取得する。
*   **本計画での位置付け:**
    *   上記は**2026年3月時点で有効性が報告されている回避策**として採用する。
    *   ただし、Netflix側のサーバーサイド仕様変更・拡張機能更新停止・DRM挙動変更により無効化される可能性があるため、**恒久保証はしない**。
*   **運用ルール（固定）:**
    1. 出力系（Xvfb / Sunshine / Moonlight）は常に `1920x1080` 固定とし、配信解像度は保証する。
    2. 元コンテンツの実効解像度（Netflix側の実配信）は都度検証し、1080p未達時は拡張機能更新・代替クライアントを適用する。
    3. 回避策が失効した場合のフォールバックとして、Windows系の公式1080p対応クライアント経路を保持する（ハイブリッド運用）。

#### 5.1.2. GPU競合（Sunshine + VOICEVOX）運用ポリシー

*   **前提:** Sunshine（NVENC）とVOICEVOX（CUDA演算）はともに `--gpus all` を利用するため、同時高負荷時にGPU使用率・VRAM使用量が競合しうる。
*   **VRAM予算（初期見積り）:**
    *   Sunshine（1080p/60 H.265）: 約1〜2GB
    *   VOICEVOX（モデルロード + 推論）: 約2〜6GB（モデル数・同時推論数で変動）
    *   余剰（OS/ドライバ・突発バースト吸収）: 2GB以上を確保する運用を推奨
*   **優先順位:** 視聴品質維持のため、同時負荷時は **Sunshineを優先**。VOICEVOX側は同時推論数を制限する。
*   **制御上の制約:** RTX 3070 Ti はMIG非対応のため、厳密分割は不可。`--gpus` 単位とアプリ側同時実行制御で運用する。
*   **監視:** `nvidia-smi` を定期取得し、GPU使用率・VRAM使用量・温度を記録。しきい値超過時はVOICEVOX負荷を段階的に落とす。

### 5.2. ゲームサーバー・公開用コンテナ群

*   **マイクラサーバー:** `itzg/minecraft-server` イメージ使用（複数コンテナで独立並立可能）
    *   CPU: `cpuset-cpus: "0-7"`（Pコア8スレッド）、メモリ上限を `mem_limit` で指定
    *   ポート割当方針: `25565`, `25566`, `25567`... のようにインスタンスごとに固定採番
    *   補足: `0-7` と `8-11` はL3キャッシュ共有のため、マイクラ高負荷時に映像配信へ軽微な干渉が出る可能性がある（要実測）。
*   **Playit.gg:** マイクラサーバーコンテナのサイドカーコンテナとして配置。ポート開放不要で外部プレイヤーをサーバーへ招待。

### 5.3. バックグラウンド処理・AIコンテナ群

*   **VOICEVOXエンジン:** 公式GPU対応Dockerイメージを使用。`--gpus all` でRTX 3070 Tiを共有し高速音声合成APIとして稼働。
    *   CPU: `cpuset-cpus: "12-15"`（Eコア）
*   **Discord Bot (Python):** Pythonカスタムイメージ。SQLiteで利用制限管理。
    *   CPU: `cpuset-cpus: "12-15"`（Eコア）

## 6. ネットワーク経路の使い分け

| 用途 | ツール | 理由 |
| :--- | :--- | :--- |
| ARグラス向け映像ストリーミング | **Tailscale** (P2P / UDP) | 低遅延が最重要。CF Tunnelはレイテンシが高くストリーミング不向き |
| Web管理画面・各種UI | **Cloudflare Tunnel** (TCP) | セキュリティ最重要。Google認証でゼロトラスト保護 |
| マイクラ等ゲームサーバー公開 | **Playit.gg** | ゲーム用プロトコル対応、自宅IP隠蔽、ポート開放不要 |

## 7. バックアップ戦略

*   **対象データ:** Dockerボリューム（マイクラのワールドデータ、Bot設定、Composeファイル群）
*   **ツール:** `rclone`（Google Drive 2TB へ差分同期）
*   **容量見積り方針:** ボリューム総量・世代数（保持日数）を事前算出し、Google Drive容量上限内に収める（2TBプラン前提）。
*   **フロー:**
    1. マイクラ稼働中バックアップは整合性確保を優先し、以下のいずれかを実施する。
       - A) `docker exec` + RCONで `/save-all` → `/save-off` 実行後にバックアップ取得、完了後 `/save-on`
       - B) `docker stop` で対象コンテナ停止 → バックアップ取得 → `docker start`
    2. 対象ボリューム（`/var/lib/docker/volumes/`等）を tar.gz 圧縮
    3. `rclone sync` でGoogle Driveへアップロード
    4. 設定ファイル（`/etc/docker/`, Composeスタック一式, `rclone.conf`, Dockgeスタック定義 `/opt/stacks/`）も別途保存
*   **復旧:** Google DriveからComposeファイル一式を取得し、`docker compose up -d` を叩くだけで全サービスが復元される設計とする。

## 8. 障害・復旧フロー

| 障害内容 | 対応手順 |
| :--- | :--- |
| コンテナ単体の異常 | DockgeのUI上でコンテナを停止 → 再デプロイ（数十秒で完結） |
| OSレベルの障害 | Ubuntu再インストール → Docker + NVIDIA Toolkit導入 → Google DriveからComposeファイル取得 → `docker compose up -d` で全復旧 |
| GPU関連エラー | ホストのNVIDIAドライババージョン確認 → 必要に応じてドライバ再インストール（コンテナには影響なし） |
